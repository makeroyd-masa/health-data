# PRD — SAM Evergreen Content Ingestion Pipelines

**Owner:** MASA / SAM
**Audience:** Claude Code (implementation agent)
**Status:** Ready to build
**Version:** 1.0

---

## 0. How to use this document

This PRD specifies a set of **content extractors / ingestion pipelines** that pull authoritative, legally-clean public content and normalize it into a single **SAM knowledge-block** format that SAM's evidence layer (and later a RAG index) will consume.

- Sections **1–5** are the **shared core**: goals, the normalized schema, the adapter interface, the pipeline, and cross-cutting (legal, caching, testing) requirements. **Read these first — they apply to every extractor.**
- Section **6** contains **one self-contained spec per use case**. You can implement them one at a time. Each names its source adapter(s), seed inputs, field mapping, tags, and acceptance criteria.
- Section **7** is repo layout and stack. Section **8** is explicitly **out of scope**.

The single most important rule: **every extractor emits the same `KnowledgeBlock` schema (§2).** Do not let schemas diverge per source. Sources are adapters behind one contract, so a future Mayo Clinic adapter (post-license) drops in without touching consumers.

---

## 1. Goals & context

SAM is an AI advocacy assistant. Its guided flows surface short, cited, plain-language content **at the point of need** (e.g., inside a "Household Readiness" flow, or when a user asks a medication question). Mayo Clinic content is being licensed for this but is weeks/months out. In the meantime we will demonstrate the mechanic — and learn which content types serve which use cases — using public-domain / open federal sources.

**In scope:** extractors for 6 use cases across 5 public sources, producing normalized knowledge blocks + a manifest, with citations, licensing metadata, caching, and change detection.

**Not in scope:** symptom triage (reserved for Mayo, post-license), the LLM router, the SAM app itself, and any Mayo content. See §8.

**Success looks like:** for each use case, running `ingest <use_case>` produces a set of valid, deduplicated, cited `KnowledgeBlock` records that the SAM prototype can query by keyword/topic and render at a point-of-need slot, with a clearly swappable source.

## 2. The normalized output: `KnowledgeBlock`

Every extractor emits records conforming to this schema. Represent it with a validated model (e.g., pydantic in Python or zod in TS). One article/page becomes **one or more** blocks — chunk by meaningful section so each block is independently surfaceable and citable.

```jsonc
{
  "id": "medlineplus:diabetes:when-to-see-a-doctor",   // stable, source-prefixed slug
  "use_case": "condition_explainer",                    // enum, see §6
  "source": "medlineplus",                              // enum: medlineplus | ready_gov | cdc | dailymed | ahrq
  "source_id": "4589",                                  // native id / setid / topic id / page slug
  "source_url": "https://medlineplus.gov/diabetes.html",
  "title": "Diabetes",
  "section": "When to see a doctor",                    // the sub-section this block represents
  "audience": "patient",                                // patient | caregiver | general
  "summary": "Short (<300 char) plain-language gist for point-of-need surfacing.",
  "body_markdown": "Clean markdown. No nav/boilerplate. Plain language.",
  "keywords": ["diabetes", "blood sugar", "hyperglycemia"],
  "codes": { "icd10": [], "snomed": [], "rxcui": [], "ndc": [] }, // populate when available
  "citation": {
    "publisher": "MedlinePlus (U.S. National Library of Medicine)",
    "source_url": "https://medlineplus.gov/diabetes.html",
    "attribution_text": "Information from MedlinePlus.gov",
    "retrieved_at": "2026-07-03T00:00:00Z"
  },
  "license": "public_domain",     // public_domain | us_gov | medlineplus_terms | ready_gov_reprint
  "source_last_updated": "2026-05-01",
  "ingested_at": "2026-07-03T00:00:00Z",
  "content_hash": "sha256:…",     // hash of normalized body; used for dedupe + change detection
  "provenance": { "extractor": "condition_explainer", "adapter": "medlineplus", "pipeline_version": "1.0" }
}
```

Notes:
- **`codes`** carry ICD-10 / SNOMED / RxCUI / NDC when the source exposes them (DailyMed and MedlinePlus Connect do). These make later matching — and swap-in of Mayo's ICD-10-indexed / HL7 Infobutton content — trivial. Populate opportunistically now.
- **`summary`** is generated by simple extraction/truncation of the section's lead — **not** an LLM call in this pipeline (keep this deterministic and offline).
- **`content_hash`** is over the normalized `body_markdown` only, so re-runs are idempotent and unchanged content is skipped.

**Output format:** write blocks as newline-delimited JSON (`out/blocks/<use_case>.jsonl`) **and** as individual markdown-with-frontmatter files (`out/markdown/<use_case>/<id>.md`) for human review. Write a top-level `out/manifest.json` (counts per use case/source, run timestamp, pipeline version, per-block id + hash + source_url).

## 3. Adapter interface

Each source is an **adapter** implementing one interface. Use-case extractors (§6) compose adapters; they do not talk to the network directly.

```python
class SourceAdapter(Protocol):
    name: str            # "medlineplus" | "ready_gov" | "cdc" | "dailymed" | "ahrq"
    license: str

    def discover(self, seed: SeedConfig) -> Iterable[SourceRef]:
        """Resolve seeds (topic lists, page URLs, drug lists, API queries) into item refs."""

    def fetch(self, ref: SourceRef) -> RawItem:
        """Fetch raw content for one ref. Uses the shared cache + rate limiter."""

    def parse(self, raw: RawItem) -> list[ParsedSection]:
        """Return cleaned sections: {section_title, body_markdown, codes?, meta}."""
```

The core pipeline turns `ParsedSection`s into `KnowledgeBlock`s (adds ids, use_case tag, citation, license, hashes, provenance) and writes output. Adapters must never emit boilerplate (site nav, cookie banners, "Skip to content," related-link rails). Convert HTML → markdown and strip chrome.

## 4. Shared pipeline requirements

1. **Config-driven seeds.** Every extractor's inputs (topic lists, page URLs, drug lists) live in `config/<use_case>.yaml`, not in code. Ship sensible starter seeds (§6) but keep them editable.
2. **Caching.** Cache every raw HTTP response to disk (keyed by URL + params) with a TTL. Re-runs read cache unless `--refresh`. This is both a courtesy to public services and what makes runs fast/idempotent.
3. **Rate limiting & retries.** Central polite HTTP client: per-host rate limits, exponential backoff with jitter (e.g., tenacity), a descriptive `User-Agent` identifying the project and a contact. **MedlinePlus caps at 85 requests/minute and asks callers to cache results for 12–24 hours — honor both.** Be conservative on all `.gov` hosts.
4. **Change detection / incremental.** Skip items whose `content_hash` and `source_last_updated` are unchanged since the last run (persist a small state file). `--refresh` forces re-fetch.
5. **Deterministic output.** Same inputs → same blocks (stable ids, sorted output). No randomness, no network calls in the transform stage, no LLM calls anywhere in this pipeline.
6. **Mandatory provenance.** Every block must have a populated `citation`, `license`, `source_url`, and `provenance`. Fail the block (log + skip, don't crash) if any is missing.
7. **CLI.** `ingest <use_case> [--refresh] [--limit N] [--dry-run]`; `ingest all`; `validate` (schema-check existing output); `stats` (print manifest summary).
8. **Logging.** Structured logs: per item fetched/cached/skipped/failed, and a run summary.

---

## 5. Cross-cutting legal & compliance requirements (non-negotiable)

These protect MASA and the eventual Mayo relationship. Bake them in.

1. **Only ingest the sanctioned feed/format for each source** (APIs, bulk files, syndication, or explicitly reproducible pages — see §6). Do not scrape around paywalls, terms, or `robots` restrictions.
2. **Attribution on every block.** Populate `citation.attribution_text`. For MedlinePlus specifically: indicate the information is from MedlinePlus.gov, **do not** use the MedlinePlus logo, and **do not** imply endorsement. For FEMA/Ready.gov reprints: **do not alter content** and **do not imply government endorsement**. Carry these as data so the UI can render them.
3. **Exclude known copyrighted subsets.** MedlinePlus mixes public-domain federal summaries with copyrighted material — **exclude the A.D.A.M. Medical Encyclopedia articles/videos and the ASHP drug monographs.** Ingest only the federally-produced health-topic summary content. If a section's attribution names a third-party rights holder, skip it and log it.
4. **License field is required** and must reflect the actual source (`public_domain`, `us_gov`, `medlineplus_terms`, `ready_gov_reprint`).
5. **No Mayo content** enters this pipeline. Mayo is a future adapter behind the same interface (§8).

## 6. Per-use-case extractor specs

Use-case enum: `household_readiness | condition_explainer | visit_prep | medication | aging_home_safety | travel_health`.

### 6.1 Household Emergency Readiness — `household_readiness`
- **Adapter:** `ready_gov` (FEMA / Ready.gov). License: `ready_gov_reprint` / `us_gov` (public domain).
- **Why this source:** For the v1 Household Readiness flow, Ready.gov is arguably a better fit than Mayo — it is public domain and publishes the exact artifacts the flow generates.
- **Access:** Ingest the sanctioned Ready.gov content pages and their linked publication PDFs (FEMA makes Ready publications free to download and reproduce under simple reprint terms). No API; a small fixed page/PDF set.
- **Seed (`config/household_readiness.yaml`):** core pages — Build A Kit, Make A Plan, Emergency Supply List, Family Emergency Communication Plan (incl. the fillable card), Evacuation, Financial Preparedness; hazard pages — Hurricanes, Floods, Wildfires, Power Outages, Extreme Heat. (Editable list.)
- **Parse/map:** each page → sections (checklist items, steps, "what to do before/during/after"). Preserve checklist structure in `body_markdown`. `audience`: `general`. Tag hazard pages with the hazard keyword so the flow's geo-risk branch can select them.
- **PDF handling:** extract text from linked publication PDFs; keep list structure; skip images/graphics (note reprint terms forbid altering — we store text as reference, we do not redistribute altered artwork).
- **Acceptance:** ≥ the full seed set ingested; supply-list and communication-plan content present as discrete blocks; each hazard page tagged; all blocks `license = ready_gov_reprint`, attribution populated, no altered content.

### 6.2 Condition explainer / new diagnosis — `condition_explainer`
- **Adapter:** `medlineplus`. License: `medlineplus_terms` (federal summaries are public domain).
- **Access:** Prefer the **MedlinePlus bulk health-topic XML files** (complete set, updated periodically) for coverage; use the **MedlinePlus Web Service** (`https://wsearch.nlm.nih.gov/ws/query?db=healthTopics&term=…`, XML) for targeted lookups/testing. Free, no registration/licensing. Respect the 85 req/min cap and 12–24h caching.
- **Seed:** either ingest all health topics from the bulk XML, or a starter list of ~50 common conditions (`config/condition_explainer.yaml`). Support both via a `mode: bulk|list` config.
- **Parse/map:** from each health-topic record, extract the **summary** and structured sub-sections. Emit blocks keyed to the sections SAM's flows need — at minimum: `Overview/Summary`, `Symptoms`, `When to see a doctor`, `Causes/Risk factors`, `Prevention` (map from the topic's groups/related content where present). **Exclude any A.D.A.M. encyclopedia content.** Capture MeSH/related codes into `codes` where present. `audience`: `patient`.
- **Acceptance:** each seeded topic yields ≥1 block; "When to see a doctor"-type guidance isolated as its own block where the source provides it; zero A.D.A.M./ASHP-attributed blocks; keyword field populated for retrieval.

### 6.3 Appointment / visit prep — `visit_prep`
- **Adapters:** `ahrq` (primary, question sets) + `medlineplus` (condition context).
- **Important:** AHRQ's Question Builder is an interactive app/website with **no API**. Ingest AHRQ's **public-domain published question content** ("Questions Are the Answer" / "Questions To Ask Your Doctor," including the "10 Questions You Should Know" and the four encounter-type question paths: talk about a health problem, get or change a medication, get medical tests, talk about surgery). Fixed, small page set.
- **Seed:** the AHRQ pages above (`config/visit_prep.yaml`) + MedlinePlus "Talking With Your Doctor" topic.
- **Parse/map:** produce **question-bank blocks** keyed by `encounter_type` (`health_problem | medication | tests | surgery | general`). Each block = a titled question set for that encounter type, plus the generic "10 questions" as a `general` block. `audience`: `patient` and `caregiver` (AHRQ content addresses caregivers explicitly — tag accordingly). Store questions as a clean markdown list in `body_markdown`.
- **Acceptance:** one block per encounter type + a general block; caregiver-tagged where applicable; all `license = us_gov` (public domain); source_url present. No dependency on any AHRQ API.

### 6.4 Medication questions — `medication`
- **Adapter:** `dailymed` (primary) + optional `medlineplus` (consumer-friendly framing).
- **Access:** **DailyMed REST API v2**, base `https://dailymed.nlm.nih.gov/dailymed/services/v2/` (JSON or XML). List/filter SPLs via `/spls` (filter by `drugname`, `rxcui`, `ndc`; paginated, 100/page max); fetch a label via `/spls/{SETID}.xml`. Public.
- **Seed:** a starter drug list — top ~100 common medications by RxCUI/name (`config/medication.yaml`), editable.
- **Parse/map:** SPL is dense clinical labeling — extract the **consumer-relevant LOINC-coded sections** and emit one block per section: Indications & Usage, Dosage & Administration, Warnings/Precautions, Adverse Reactions (side effects), Drug Interactions, and the patient-facing sections (Patient Package Insert / "Information for Patients") when present. Populate `codes.rxcui` / `codes.ndc` / `setid`. Set `audience`: `patient`. Where SPL language is highly technical, keep it faithful (do **not** rewrite medical content) but store the patient-facing section as the preferred surfacing block.
- **Note:** DailyMed drug labels are authoritative but clinical; this is exactly a spot where licensed Mayo consumer drug content would later improve readability — the swap is a source change, same schema.
- **Acceptance:** each seeded drug resolves to ≥1 SPL and yields section blocks; `codes.rxcui` populated; patient-facing section flagged where available; idempotent on re-run by SETID + version.

### 6.5 Aging / caregiving / home safety — `aging_home_safety`
- **Adapters:** `medlineplus` (primary) + `cdc` (STEADI falls-prevention, via syndication).
- **Access:** MedlinePlus topics as in §6.2. **CDC Content Syndication API**: `https://tools.cdc.gov/api/v2/resources/media` (JSON/XML); enumerate via `/topics` and `/resources/media`, fetch cleaned HTML via `/resources/media/{id}/syndicate` (scripts stripped by default). Note: not all CDC content is syndicated; filter by topic and by `sourceUrl` prefix.
- **Seed:** MedlinePlus topics — Falls, Home Safety, Caregivers, Assistive Devices, Older Adult Health, Alzheimer's Caregivers (`config/aging_home_safety.yaml`); CDC — STEADI / Older Adult Fall Prevention media items. (Optional future: NIA content via the HHS Digital Media syndication platform — leave a stub; confirm availability before enabling.)
- **Parse/map:** home-safety and fall-risk content → checklist/room-by-room blocks where the source provides them; caregiver content tagged `audience: caregiver`. Preserve list structure.
- **Acceptance:** ≥ seed set ingested across both adapters; caregiver vs. patient audience tagged; CDC blocks carry CDC attribution + `us_gov` license; MedlinePlus exclusions (A.D.A.M.) respected.

### 6.6 Travel health readiness — `travel_health`
- **Adapter:** `cdc` (CDC Travelers' Health, via syndication).
- **Access:** CDC Content Syndication API (as in §6.5). Filter for Travelers' Health media (e.g., topic = travel; `sourceUrl` under the CDC travel site). Fetch cleaned HTML via `/syndicate`.
- **Seed:** general travel-prep content (before-you-go checklists, travel health kit, "getting sick/injured abroad," destination-prep guidance) (`config/travel_health.yaml`). **Exclude time-sensitive travel *notices*** (they're dynamic and would stale quickly) unless a `include_notices: true` flag is set.
- **Parse/map:** prep checklists and "what to do if you need care abroad" → blocks; tag with `travel`. This use case connects to SAM's travel branch / benefits, so keep "care away from home" content as its own block. `audience`: `general`.
- **Acceptance:** ≥ seed set ingested; notices excluded by default; CDC attribution + `us_gov` license on every block.

## 7. Repo layout & stack

Default stack: **Python 3.11+** (`httpx` or `requests`, `lxml`/`beautifulsoup4`, `markdownify`, `pydantic` v2, `tenacity`, `pyyaml`, `orjson`). TypeScript/Node is acceptable if preferred — keep the schema and interface identical.

```
sam-content-ingest/
  config/
    household_readiness.yaml
    condition_explainer.yaml
    visit_prep.yaml
    medication.yaml
    aging_home_safety.yaml
    travel_health.yaml
  src/
    core/
      schema.py          # KnowledgeBlock model + validation
      pipeline.py        # section -> block, hashing, dedupe, write
      http.py            # polite client: cache, rate limit, retries, UA
      cache.py
      cite.py            # attribution/license helpers
      chunk.py           # section splitting, html->markdown, boilerplate strip
      cli.py             # ingest / validate / stats
    adapters/
      medlineplus.py
      ready_gov.py
      cdc_syndication.py
      dailymed.py
      ahrq.py
    extractors/
      household_readiness.py
      condition_explainer.py
      visit_prep.py
      medication.py
      aging_home_safety.py
      travel_health.py
  out/
    blocks/*.jsonl
    markdown/<use_case>/*.md
    manifest.json
  tests/
    fixtures/            # recorded API/page responses (do NOT hit live services in CI)
    test_<adapter>.py
    test_schema.py
  README.md
```

**Testing:** unit-test each adapter's `parse` against **recorded fixtures** (never hammer live `.gov` services in CI). Validate all emitted blocks against the schema. Include one end-to-end test per use case that runs the extractor against fixtures and asserts the acceptance criteria in §6.

## 8. Out of scope (explicit)

- **Symptom triage / "should I be concerned?" logic.** Reserved for **Mayo Clinic** licensed content (their evidence-based symptom-triage algorithms) once licensed. Do **not** build or approximate triage here, and do not ingest any triage content.
- **Any Mayo Clinic content.** When the license lands, add a `mayo.py` adapter implementing the same `SourceAdapter` interface (their content is available via Bulk/Realtime API, ICD-10-indexed, HL7 Infobutton-ready) and re-run the same extractors. The `codes` field and schema already anticipate this — no consumer changes required.
- **The LLM router, RAG index build, and the SAM app UI.** This pipeline produces the knowledge blocks those systems consume; it does not implement them. Output must be a clean, queryable block set + manifest.

---

## 9. Definition of done

- `ingest all` produces valid, deduplicated, cited blocks for all six use cases, plus `manifest.json`.
- Every block passes schema validation and carries `citation`, `license`, `source_url`, `provenance`.
- Re-running without `--refresh` performs no network calls (cache hits) and produces byte-identical output.
- No A.D.A.M./ASHP content; no altered FEMA reprints; MedlinePlus attribution rules respected; no Mayo content.
- Tests pass against fixtures; a short `README.md` documents each source, its access method, its license, and how to run each extractor.
```
